\section{Platform Overview}

SOCIO is a fully production-deployed, multi-product university event management platform. It digitises the complete lifecycle of university events — from creation and multi-authority approval through student registration, ticketing, on-ground attendance scanning, post-event analytics, and physical campus access control.

The platform is currently live at Christ University, Bengaluru, serving real students, organisers, department heads, and administrative staff. It is built with modern, scalable technology and is architected to support multiple university campuses simultaneously.

\subsection{User Role System}

SOCIO implements a 10+ role access control system with time-bound role assignment:

\begin{multicols}{2}
\begin{itemize}[leftmargin=*]
    \item \textbf{Student} — default registered user
    \item \textbf{Organiser} — can create and manage events/fests/clubs
    \item \textbf{Volunteer} — assigned to specific events for attendance scanning
    \item \textbf{HOD} — department approval authority (blocking stage)
    \item \textbf{Dean} — institution-level approval authority (blocking stage)
    \item \textbf{CFO} — financial approval (optional stage)
    \item \textbf{Accounts} — finance office sign-off (optional stage)
    \item \textbf{Venue Manager} — handles venue booking requests
    \item \textbf{Catering Vendor} — handles catering orders
    \item \textbf{Stalls Coordinator} — handles stall bookings
    \item \textbf{IT Support} — handles IT request queue
    \item \textbf{Master Admin} — full platform control
\end{itemize}
\end{multicols}

Role assignments include expiry dates, enforced server-side on every authenticated request.

\subsection{Approval Workflow Engine}

A complete multi-stage institutional event approval system is implemented:

\begin{infobox}
\textbf{Approval Flow:}\\[0.3cm]
\textbf{Stage 1 (Blocking):} HOD Approval \quad$\rightarrow$\quad \textbf{Stage 2 (Blocking):} Dean Approval\\[0.2cm]
\textbf{Stage 3 (Optional):} CFO Approval \quad$\rightarrow$\quad \textbf{Stage 4 (Optional):} Finance/Accounts\\[0.2cm]
\textbf{Parallel (Non-blocking):} IT Request \textbullet\ Venue Booking \textbullet\ Catering \textbullet\ Stalls\\[0.3cm]
Once all blocking stages are approved or skipped, the event goes live (\texttt{draft = false}).
\end{infobox}

Features of the approval engine:
\begin{itemize}[leftmargin=*]
    \item Full audit log per stage with timestamps and approver email
    \item Fest events inherit approval from parent fest (bypass blocking stages)
    \item Budget line item tracking per event (quantity × unit price)
    \item Each role-holder sees only their own queue
    \item Revision notes supported at each stage
\end{itemize}

\subsection{Native and PWA Features}

\begin{itemize}[leftmargin=*]
    \item \textbf{Shake-to-Scan:} Accelerometer via \texttt{@capacitor/motion} — shake the device to open QR scanner instantly
    \item \textbf{Haptic Feedback:} Distinct haptic patterns for successful scan vs.\ error scan
    \item \textbf{Offline QR Queue:} Scans made without internet are queued locally and submitted automatically on reconnect
    \item \textbf{Native MLKit Barcode Scanning:} Hardware-accelerated on Android; ZXing fallback in web/PWA mode
    \item \textbf{Orientation Lock:} Scanner screen locks to landscape for optimal scanning
    \item \textbf{Desktop Gate:} Component blocks non-mobile access to app-specific pages
    \item \textbf{PWA Install Prompt:} Smart Add-to-Home-Screen prompt
    \item \textbf{Smart Notification Prompt:} Context-aware push permission request
    \item \textbf{Session Persistence:} localStorage-based session restoration for PWA without re-authentication
    \item \textbf{Onboarding Overlay:} Native onboarding flow on first app launch
    \item \textbf{Campus Selector:} Multi-campus filtering on discovery screens

\section{SOCIOGATED — Campus Gate Access Management}

\subsection{Overview}

SOCIO Gated is a standalone gate access control system designed for campus security officers and event organisers. It manages physical entry to campus events — handling visitor registrations, QR pass generation, gate-level verification, and entry approvals. It runs on a completely separate database from the main platform for security isolation, while maintaining bi-directional sync with SOCIO Web.

\subsection{Integration with SOCIO Web}

The \texttt{gatedSync.js} module in the socioweb server maintains bi-directional sync:

\begin{itemize}[leftmargin=*]
    \item Events and fests created in SOCIO Web are automatically pushed to Gated's \texttt{event\_requests} table
    \item Outsider registrations created in SOCIO Web automatically create visitor records in the Gated database
    \item Gated organiser accounts are auto-provisioned from SOCIO organiser email addresses
\end{itemize}
